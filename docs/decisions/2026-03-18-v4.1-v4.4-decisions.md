# Decision Log: v4.1.0 → v4.4.0

**Date:** 2026-03-18
**Scope:** ScholarGraph3D v4.1.0 through v4.4.0
**Summary:** Major architectural decisions around gap visualization, layout engine, bridge detection, and UX simplification.

---

## 1. Gap Visualization Approach

**Context:** The initial implementation (v4.1) rendered research gaps directly in the 3D scene using arcs between clusters, brief summary cards floating in 3D space, and highlighted "foundation" nodes. The goal was to make gaps visually prominent within the graph itself.

**Options considered:**
- **(A) Full 3D overlay** — Gap arcs, void regions, foundation node highlights, distance labels rendered in Three.js.
- **(B) Camera-only + sidebar** — Move the camera to frame the relevant clusters and display gap information in a 2D sidebar panel (Field Relationship View).
- **(C) Hybrid** — Keep minimal 3D indicators (e.g., dotted lines) with detailed info in a sidebar.

**Decision:** Option B — camera-only navigation with a sidebar Field Relationship View.

**Rationale:** 3D overlays (arcs, floating cards, distance labels) added substantial rendering complexity and visual clutter without proportional informational value. Users had difficulty reading text and interpreting spatial cues in a rotating 3D scene. A 2D sidebar with structured data cards proved far more readable, and camera-based framing still communicates spatial relationships between clusters. This also eliminated an entire class of z-fighting and occlusion bugs.

**Commits:** `a95b7c1`, `7a06c73`, `ba48b77`

## 2. Field Relationship Classification

**Context:** When a user selects two clusters to compare, the system needs to characterize the relationship between them. A simple "similar / different" binary was insufficient — users needed actionable insight about where opportunities lie.

**Options considered:**
- **(A) Binary classification** — "Related" vs. "Unrelated" based on a single similarity threshold.
- **(B) Four-pattern taxonomy** based on two independent dimensions: centroid_similarity and citation_density.
- **(C) LLM-generated narrative description** of the relationship.

**Decision:** Option B — four patterns derived from centroid_similarity x citation_density:

| Pattern | Similarity | Citation Density | Interpretation |
|---|---|---|---|
| **Unknown Neighbors** | High | Low | Most actionable gap — fields are semantically close but researchers are not citing across them. |
| **Emerging Bridge** | Moderate | Low-Moderate | Early cross-pollination; bridge papers exist but are sparse. |
| **Different Fields** | Low | Low | Genuinely separate domains; gap is expected. |
| **Well Connected** | Any | High | Already integrated; no intervention needed. |

**Rationale:** The two-axis model is transparent and reproducible (no LLM variance). "Unknown Neighbors" directly surfaces the highest-value research opportunities — fields that should be talking to each other but are not. LLM narration (Option C) was rejected because it introduces latency, cost, and non-determinism for a classification that can be computed deterministically.

**Commits:** `a95b7c1`

## 3. Force-Directed Layout (Replacing Fixed UMAP Positions)

**Context:** The original layout pinned every node to its UMAP-projected (x, y) coordinates with `fx`/`fy`/`fz` set, using publication year mapped to the Z-axis. This produced readable but rigid layouts.

**Options considered:**
- **(A) Keep fixed UMAP positions** — stable, reproducible, but clusters appear as linear streaks along the time axis.
- **(B) UMAP as seed, then free force-directed simulation** — use UMAP coordinates as initial positions, then let d3-force-3d settle the graph with citation link forces and cluster gravity.
- **(C) Pure force-directed** — ignore UMAP entirely, start from random positions.

**Decision:** Option B — UMAP-seeded force-directed layout with cluster gravity.

**Rationale:** Fixed UMAP positions caused clusters to appear as elongated linear formations (especially along Z = time), which made it hard to perceive cluster boundaries. Removing `fx`/`fy`/`fz` and adding a custom cluster gravity force allowed the simulation to naturally separate clusters into distinct "island" formations. UMAP seeding ensures the initial configuration is already reasonable, so the simulation converges quickly. Tuned `cooldownTicks` and `velocityDecay` to balance stability with responsiveness.

**Commits:** `47eab24`

## 4. Bridge Paper Detection (Similarity → Citation-Based Hybrid)

**Context:** Bridge papers connect two otherwise separate research clusters. The initial implementation scored candidates purely on embedding similarity — geometric mean of cosine similarity to both cluster centroids.

**Options considered:**
- **(A) Pure embedding similarity** — geometric mean of cosine sim to both centroids. Simple but prone to false positives (a paper can be topically between two fields without actually bridging them).
- **(B) Pure citation evidence** — count how many papers in each cluster cite or are cited by the candidate. Robust but misses papers that are topically bridging without yet being cited.
- **(C) Weighted hybrid** — 70% citation evidence + 30% embedding similarity.

**Decision:** Option C — citation-based hybrid scoring with 70/30 weighting. Added `cited_by_a_count` and `cited_by_b_count` fields to the bridge paper data model.

**Rationale:** Pure similarity (Option A) generated too many false positives — papers positioned in embedding space between two clusters but with no actual citation links to either. Citation evidence is a much stronger signal that a paper genuinely serves as an intellectual bridge. The 30% similarity component retains the ability to surface new bridge candidates that have not yet accumulated citations. The per-cluster citation counts (`cited_by_a_count`, `cited_by_b_count`) let the UI show evidence strength to the user.

**Commits:** `c697a7d`

## 5. Similarity Edge Hover Tooltip (LLM-Free)

**Context:** Similarity edges (dashed lines between semantically similar papers across clusters) needed hover tooltips explaining why two papers are linked. An earlier design considered LLM-generated explanations.

**Options considered:**
- **(A) LLM-generated explanation** — send both abstracts to an LLM and get a natural-language similarity summary. Rich but slow and expensive.
- **(B) LLM-free extraction** — shared keywords from title/abstract word intersection, citation direction if one cites the other, shared field labels.
- **(C) No tooltip** — rely on the edge itself as sufficient information.

**Decision:** Option B — LLM-free tooltip with shared keywords, citation direction, and shared fields.

**Implementation detail:** Three.js raycasting against thin line geometries proved unreliable. Instead, implemented mouse proximity detection by projecting edge endpoints to screen coordinates via `graph2ScreenCoords`, then computing point-to-segment distance (`distToSegment`). This is computed every mouse move and triggers the tooltip when the cursor is within a pixel threshold of any similarity edge.

**Rationale:** LLM calls per hover event are impractical (latency and cost). The extracted shared keywords, citation relationship, and field overlap provide sufficient context for users to understand the connection. Screen-space proximity detection is more reliable than raycasting for thin lines and avoids the need for invisible hit-area geometries (which were attempted and reverted).

**Commits:** `2f41fc3`, `c9a28a3`, `c265b07`, `beb8a1a`, `9da3d31`, `f4beba4`, `f89c029`, `8286b69`

## 6. Directional EXPAND

**Context:** The original "EXPAND NETWORK" button fetched a mix of references and citations up to a shared limit, giving the user no control over expansion direction.

**Options considered:**
- **(A) Single button, shared limit** — "EXPAND NETWORK (50)" fetches up to 25 refs + 25 cites.
- **(B) Two directional buttons** — "REFS (N) →" and "← CITES (M)" with independent full limits per direction.
- **(C) Dropdown with direction filter** — single button with a direction selector.

**Decision:** Option B — two separate buttons with full limit per direction.

**Rationale:** Researchers often want to explore a paper's intellectual lineage (references) or its impact (citations) independently. Splitting the limit (Option A) meant each direction got only half the budget. With directional expand, each direction gets the full limit (e.g., 50 refs OR 50 cites). The two-button UI is also more discoverable than a dropdown and communicates available counts at a glance.

**Commits:** `375a5c9`, `d1d5e5e`

## 7. Removed Features

**Context:** Several features were removed during the v4.1 → v4.4 cycle as part of a complexity reduction effort.

| Feature | Reason for removal |
|---|---|
| **Gap overlay (132 distance label)** | Floating numeric labels in 3D space were hard to read, occluded by nodes, and the absolute distance value was not meaningful to users without calibration context. |
| **Research Questions generation** | LLM-generated research questions from gap analysis added latency and were often generic. Users preferred to interpret the Field Relationship View themselves. |
| **3D gap arc / void / foundation highlights** | Replaced by camera-only + sidebar approach (see Decision 1). |
| **Node drag** | Dragging individual nodes in a force-directed layout caused cascading repositioning of neighbors, creating a disorienting experience. The force simulation handles positioning better without manual intervention. |

**Rationale (general):** Each removed feature had a specific issue, but the overarching principle was that complexity must earn its place. Features that added visual noise, LLM latency, or interaction confusion without clear user benefit were cut.

**Commits:** `de6b455`, `7a06c73`

## 8. Nebula Opacity

**Context:** Cluster "nebula" effects (particle clouds and rings around cluster boundaries) were nearly invisible at the original opacity values, defeating their purpose of showing cluster extent.

**Options considered:**
- **(A) Keep subtle (0.06 particles, 0.012 ring)** — minimalist but users did not notice the clusters.
- **(B) Increase to visible range (0.15 particles, 0.10 ring)** — clearly visible without obscuring nodes.
- **(C) Replace with solid convex hull rendering** — unambiguous boundaries but visually heavy and occludes interior nodes.

**Decision:** Option B — particles at 0.15 opacity, rings at 0.10.

**Rationale:** The original values were too subtle for the nebula to serve its purpose (communicating cluster boundaries). The increased values make clusters perceptible at a glance while remaining translucent enough not to obscure the nodes and edges they contain. Convex hulls (Option C) were rejected as too visually dominant for an ambient cue.

**Commits:** `de6b455`

## 9. Cluster Gravity Edge Case (Single Cluster)

**Context:** The custom cluster gravity force pulls each node toward its cluster centroid. When the dataset contains only one cluster, all nodes share a single centroid and the force collapses the entire graph into a tight ball.

**Options considered:**
- **(A) Always apply cluster gravity** — simple but breaks single-cluster datasets.
- **(B) Disable cluster gravity when cluster count = 1** — let the standard repulsion and link forces handle layout.
- **(C) Scale gravity strength inversely with cluster count** — smoother degradation but added complexity.

**Decision:** Option B — disable cluster gravity entirely when only one cluster is detected.

**Rationale:** With a single cluster, there is no inter-cluster separation to maintain, so cluster gravity has no useful purpose. Disabling it lets the existing repulsion and link forces produce a readable layout. Option C was over-engineered for an edge case.

**Commits:** `9b8d325`

## 10. Similarity Edge Link Force Strength

**Context:** Similarity edges (connecting semantically similar papers across clusters) were treated with the same link force strength as citation edges, causing cross-community edges to pull distant clusters together and compress the overall layout.

**Options considered:**
- **(A) Same force as citations (0.4)** — similarity edges distort cluster separation.
- **(B) Near-zero force (0.01)** — edges are rendered but exert negligible layout influence.
- **(C) Remove similarity edges from the force simulation entirely** — render as visual-only overlays.

**Decision:** Option B — similarity edge link force set to 0.01 (vs. 0.4 for citations).

**Rationale:** Similarity edges serve an informational purpose (showing cross-cluster connections) but should not override the cluster structure established by citations and cluster gravity. A force of 0.01 keeps them in the simulation (so d3-force-3d is aware of them) without meaningfully distorting the layout. Option C (visual-only) would have required maintaining a separate edge rendering pipeline outside the force graph.

**Commits:** `0132f35`
